# Data Dictionary & Understanding: Customer Support Tickets

This document describes the schema, structure, constraints, and data quality issues of the raw dataset: `customer_support_tickets.csv`.

---

## 1. Schema Definitions

The dataset contains **2,505 rows** (including header and duplicates) and **14 columns**.

| Column Name | Data Type (CSV) | DB Type (MySQL) | Nullable? | Key Type | Description |
| :--- | :--- | :--- | :--- | :--- | :--- |
| **Ticket_ID** | String | `VARCHAR(10)` | No | Primary Key | Unique identifier for each ticket (format: `TS-XXXX`). |
| **Customer_ID** | String | `VARCHAR(10)` | No | Foreign Key | References the Customer who raised the ticket (format: `CU-XXXX`). |
| **Agent_Name** | String | `VARCHAR(50)` | No | Attribute | The support agent assigned to resolve the issue. |
| **Department** | String | `VARCHAR(20)` | No | Attribute | The organizational unit handling the ticket. |
| **Priority** | String | `VARCHAR(10)` | No | Attribute | Priority level (`Low`, `Medium`, `High`, `Urgent`). |
| **Category** | String | `VARCHAR(50)` | No | Attribute | The specific type of request. |
| **Created_Date** | DateTime | `DATETIME` | No | Attribute | The date/time when the ticket was opened. |
| **Closed_Date** | DateTime | `DATETIME` | Yes | Attribute | The date/time when the ticket was closed. Null for open tickets. |
| **Resolution_Time_Hours**| Decimal | `DECIMAL(6,2)` | Yes | Attribute | Total duration in decimal hours from creation to closure. |
| **SLA_Status** | String | `VARCHAR(10)` | No | Attribute | Indicates if the SLA was `Met` or `Breached` (based on priority SLA hours). |
| **CSAT_Score** | Integer | `TINYINT` | Yes | Attribute | Customer rating of the service (1 to 5). Blank if survey unreturned. |
| **Region** | String | `VARCHAR(20)` | No | Attribute | Geography of the customer. |
| **Channel** | String | `VARCHAR(15)` | No | Attribute | Ticket intake channel (`Email`, `Chat`, `Phone`, `Portal`). |
| **Status** | String | `VARCHAR(15)` | No | Attribute | Current ticket lifecycle status (`Open`, `In Progress`, `Pending`, `Closed`). |

---

## 2. Relational Database Key Architecture
* **Primary Key (PK)**: `Ticket_ID`. In a clean transactional state, this must be a non-null, completely unique field.
* **Foreign Keys (FK)**: 
  * `Customer_ID`: In a full corporate CRM, this links to a separate `Customers` dimension table.
  * `Agent_Name` / `Agent_ID`: In a standard database, agent names would map to a separate `Agents` dimension table. For this project, we are keeping it as a denormalized attribute to simplify initial modeling.

## 3. Deliberate Data Quality Issues & Cleaning Plan

To simulate real-world data profiling work, the dataset contains specific anomalies that must be addressed before analysis:

### A. Missing Values (Nulls)
* **CSAT_Score (Numeric Nulls)**: 
  * *Reason*: Customers are not forced to complete surveys. Also, tickets that are still open (`Open`, `In Progress`, `Pending`) cannot have CSAT scores.
  * *Volume*: ~20% of closed tickets are missing CSAT. All open tickets (approx. 12% of total dataset) are missing CSAT.
* **Closed_Date & Resolution_Time_Hours (Temporal/Numeric Nulls)**:
  * *Reason*: Tickets that are currently `Open`, `In Progress`, or `Pending` have not yet been resolved.
  * *Volume*: Approx. 12% of records.

### B. Duplicate Rows
* *Anomaly*: **5 complete duplicate rows** exist in the CSV (e.g. duplicate logs generated by database write glitches).
* *Impact*: Left untreated, these duplicates will inflate counts (e.g. Total Tickets) and skew average resolution times.
* *Plan*: Identify and drop these duplicates during the SQL and Excel phases.

### C. Text Typos (Structural Inconsistencies)
* *Agent Name Typos*: 
  * `"Emily Davis"` is sometimes logged as `"Emly Davis"`.
  * `"David Lee"` is sometimes logged as `"David L."`.
* *Category Typos*:
  * `"Payment Failure"` is sometimes logged as `"Paymnt Failure"`.
  * `"Reset Password"` is sometimes logged as `"Resst Password"`.
  * `"Subscription Cancel"` is sometimes logged as `"Subscrpt Cancel"`.
* *Impact*: In BI reports, these will generate separate chart bars (e.g. a small slice for "Paymnt Failure" and a large one for "Payment Failure"), ruining aggregation summaries.
* *Plan*: Correct typos using standardizing queries in SQL or manual replacements/conditional values in Excel.
